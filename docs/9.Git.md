## Git

## 기본 명령어

### 커밋

`git add`: stage에 변경된 파일을 staging함  
`git status`: file의 staging 상태를 확인

`git diff` : Committed(Staged) vs Modified  
`git diff --staged` : Committed vs Staged

`git commit -m "commit message"`: 커밋 (-m; 메시지와 함께 커밋)

- `git commit --amend`: 가장 마지막커밋(HEAD)을 새로운 커밋에 바로 반영(신규커밋으로 대체)
- `git commit --fixup`: 신규 커밋이 HEAD에 대한 fixup 커밋으로 생성됨. 기존 HEAD는 그대로 유지 -> 그 위에 fixup commit이 새로 생김(HEAD)

`git restore  --staged [<pathspec>]` : 파일의 변경을 Staging area에서 삭제. Staged → Untracked ↔ `git add`

- `git reset HEAD [<pathspec>]`

`git clean` : Staging area에 반영되지 않은 working tree에서의 변경사항을 버림

- `git restore [<pathspec>]`

### 커밋 메세지 컨벤션

```md
feat : 새로운 기능 추가  
fix : 버그 수정  
docs : 문서 수정  
style : 코드 포맷팅, 세미콜론 누락, 코드 변경이 없는 경우  
refactor : 코드 리펙토링  
test : 테스트 코드, 리펙토링 테스트 코드 추가  
chore : 빌드 업무 수정, 패키지 매니저 수정
```

### 브랜치

**`HEAD` : 현재 체크아웃 된 커밋 = 현재 작업중인 커밋**

- HEAD는 항상 작업트리의 가장 최근 커밋을 가리킨다.
- 커밋 히스토리와 작업 영역에 영향을 주는 git 명령어들은 대부분 HEAD를 변경한다.
- HEAD가 있는 곳 다음에 새로운 커밋이 추가된다.

`git branch 브랜치명`: 브랜치 생성  
`git checkout 브랜치명`: 브랜치 변경
`git merge 브랜치명`: 현재 브랜치에 다른 브랜치의 커밋을 불러와 병합

`git stash` : working tree의 내용을 Staging area가 아닌 다른 임시 stack에 저장
`git stash pop`: 가장 최근에 한 stash를 적용하고 stash에서 삭제

### 원격

`git remote` : 원격 저장소와 관련된 설정

- `git remote -v`
- `git remote add` : `origin` 이라는 별명은 여기서 설정하는 것
- `git remote remove`

`git clone`

- 원격 저장소의 내용을 복제하여, 로컬의 현재 경로에 새로운 저장소를 생성

---

`git push`

- 로컬 브랜치의 내용을 원격 서버로 전송
  `git push origin [<branch name>]`

`git fetch`

- 서버에 존재하지만 로컬에 없는 데이터를 받아와 저장함.
  `git fetch origin`
- 워킹 디렉토리의 파일 내용은 변경되지 않고, `git merge`를 해야 합쳐진다.

`git pull`

- `fetch` + `merge`

### 커밋 수정

### Tip!

- SHA-1 해시값으로 되어있는 커밋, 전체가 아닌 일부만 사용해도 된다`commit 8984de38057a0cd157f978ff16d8a56e53fc9d53` = `8984d`
- 한번에 한 커밋 위로 움직이는 `^` : `git checkout main^`
- 한번에 여러 커밋 위로 움직이는 `~<num>` : `git checkout main~1`

`git revert` : commit의 결과를 원복하는 revert commit 생성

- 이미 원격에 공유되어 타인이 사용하고 있을 가능성이 있는 commit은 명시적으로 revert commit을 생성하여 원복한다.
- 예) 이미 배포된 커밋에 문제가 있어서 원복 해야할 경우

`git reset` : HEAD의 위치를 옮김. 마치 커밋을 하지 않았던 것 처럼..

- `git reset --soft HEAD~`:
- `git reset [--mixed] HEAD~`:
- `git reset --hard HEAD~`:

**`git cherry-pick` : 필요한 커밋만 고르기.** HEAD 아래에 다른 커밋들에 대한 복사본을 만듦

- 커밋 정리가 점점 복잡해 질 때 아주 유용한 명령어.
- 예) `feature-login` 브랜치에 로그인 관련 커밋을 20개 추가했는데, 이 중 15개만 쓸모가 있어서 브랜치를 정리하고 싶은 경우
- `git cherry-pick <commit1> <commit2> ....`

`git rebase -i` : interactive rebase

- HEAD로 부터 커밋 범위를 정하여 해당 커밋들을 재작성(수정,삭제) 할 수 있다.
- rebase -i 는 편집기에서 커밋을 선택할 수 있기 때문에 범위만 알아도 수정이 가능

> Tip
>
> 1. 남들과 공유하는 커밋은 rebase하지 말자. 나의 로컬에서만 한다고 생각하자.
> 2. git pull

## gitignore

`.gitignore` 파일 생성

```md
# 파일 제외 (파일명.확장자)

파일명.txt

# 현재 경로에 있는 파일만 제외 (다른 경로의 동일한 파일명은 추적)

/파일명.txt

# 특정 경로안의 특정 파일 제외

폴더명/파일명.txt

# 특정 폴더안의 파일 전부 제외

폴더명/

# 해당 확장자 파일 전체 제외

\*.txt

# 예외

!제외할 파일명.txt
```
